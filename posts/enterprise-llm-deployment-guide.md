# 企业大模型部署实战：从硬件选型到落地交付的全链路指南

> "我要在企业里用大模型，但不知道从哪开始。"——这可能是 2025 年被问到最多的问题。本文从硬件选型、架构设计、技术路线、成本估算到真实案例，给你一份可以直接落地的参考。

---

## 一、先回答一个问题：企业为什么需要自己部署？

### 三种路线，三种逻辑

| 路线 | 适合谁 | 核心逻辑 |
|------|-------|---------|
| **直接调 API** | 快速验证、非敏感场景 | 成本低、上手快，但数据出境、定制有限 |
| **私有化部署** | 金融/政务/医疗/军工等强监管行业 | 数据不出域、完全可控，但投入大 |
| **混合方案** | 大多数成长期企业 | 核心数据走私有模型，通用能力调 API |

> **核心判断标准**：数据是否可以出域？如果答案是否，私有化部署是唯一选项。如果能出域但调用量大，混合方案是性价比最优解。

## 二、硬件选型：算力是地基

### GPU 选型速查表

| GPU | 显存 | FP16算力 | 单卡价格(约) | 定位 |
|-----|------|---------|------------|------|
| **RTX 4090** | 24GB | 165 TFLOPS | ¥1.3万 | 消费级性价比之王，7B-14B推理首选 |
| **L40S** | 48GB | 362 TFLOPS | ¥5万 | 推理+图形渲染，中等规模部署 |
| **A800 80GB** | 80GB | ~312 TFLOPS | ¥7万 | 国内市场专供，训练+推理通用 |
| **H20** | 96/141GB | 148 TFLOPS | ¥6-8万 | 当前国内可售专用AI芯片，大显存优势 |
| **A100 80GB** | 80GB | 312 TFLOPS | ¥12-15万 | 训练+推理基石 |
| **H800 80GB** | 80GB | ~990 TFLOPS | ¥6万(受限) | 国内替代H100，算力受限但性价比高 |
| **H100 80GB** | 80GB | 990 TFLOPS | ¥26-30万 | 大规模训练王者 |

> **选型原则**：推理优先看显存（模型能不能装下），训练优先看算力（能不能快速迭代）。受出口管制影响，国内企业优先考虑 A800/H20/昇腾910B。

### 不同模型规模的硬件需求

| 模型规模 | 量化方式 | 显存需求 | 推荐配置 | 典型模型 |
|---------|---------|---------|---------|---------|
| **1.5-4B** | FP16 | 4-8GB | 1× RTX 3060 | Qwen2.5-1.5B, Phi-4 |
| **7-8B** | INT4 / FP16 | 8GB / 16GB | 1× RTX 4090 | Qwen2.5-7B, Llama3-8B |
| **14B** | INT4 / FP16 | 16GB / 28GB+ | 1× RTX 4090 / 2× A100 | Qwen2.5-14B |
| **32-34B** | INT4多卡 / FP16多卡 | 24GB+ / 64GB+ | 2× RTX 4090 / 2× A100 | Qwen2.5-32B, CodeLlama-34B |
| **70-72B** | INT4 / FP16 | 48GB(4卡) / 160GB+ | 4× A100 / 8× A100 | Qwen2.5-72B, Llama3-70B |
| **671B** | FP8多机 | ~1.3TB+ | 2台8×A100集群 | DeepSeek-V3 |

> **关键计算**：FP16 精度下，模型显存 ≈ 参数量 × 2 字节。7B 模型约需 14GB，INT4 量化后约 4GB。实际推理还需额外 20-30% 用于 KV Cache 和运行时开销。

| 厂商 | 代表型号 | 算力 | 显存 | 生态适配 | 单卡预估价 | 定位 |
|------|---------|------|------|---------|-----------|------|
| **华为昇腾** | 910B / 910C | 320/370+ TFLOPS | 64/80GB+ | 极高（CANN+PyTorch+MindSpore） | ¥10-12万 | 国产替代首选 |
| **寒武纪** | 思元590 | 256 TFLOPS | 64GB | 中等偏上 | ¥5-8万 | 推理+中小训练 |

> **最新动态**：DeepSeek 已全面适配华为昇腾，在昇腾上单卡推理性能达 H20 的 2.87 倍。对于信创要求的企业，昇腾 + DeepSeek 是当前最成熟的国产化方案。

## 三、架构设计：决定系统的天花板

### 整体架构图

```
┌──────────────────────────────────────────────────────────┐
│                      应用层                               │
│   智能客服 · 知识问答 · 代码助手 · 文档生成 · 数据分析     │
├──────────────────────────────────────────────────────────┤
│                     编排层                                │
│   Agent框架(LangChain/Dify) · 提示工程 · 工作流引擎       │
├──────────────┬───────────────────────┬───────────────────┤
│   模型服务层  │     知识增强层         │    数据管理层      │
│  vLLM/TGI   │   RAG Pipeline       │   数据采集         │
│  Ollama     │   向量数据库          │   数据清洗         │
│  推理优化    │   Embedding模型       │   标注管理         │
├──────────────┴───────────────────────┴───────────────────┤
│                     基础设施层                             │
│   GPU集群 · K8s容器编排 · 监控告警 · 日志 · CI/CD         │
└──────────────────────────────────────────────────────────┘
```

### RAG 架构：企业大模型的核心能力

绝大多数企业场景不需要"训练大模型"，而是需要"让大模型理解企业知识"——这就是 RAG（检索增强生成）的核心价值。

```
用户提问
   │
   ▼
┌─────────┐    ┌──────────────┐    ┌──────────────┐
│ Query   │───→│  Embedding   │───→│  向量数据库   │
│ 改写/扩展│    │  模型        │    │  检索Top-K   │
└─────────┘    └──────────────┘    └──────┬───────┘
                                          │
                                          ▼
                                   ┌──────────────┐
                                   │  重排序(Rerank)│
                                   │  去重·截断     │
                                   └──────┬───────┘
                                          │
                                          ▼
                                   ┌──────────────┐
                                   │  Prompt组装   │
                                   │  系统提示+     │
                                   │  检索结果+问题 │
                                   └──────┬───────┘
                                          │
                                          ▼
                                   ┌──────────────┐
                                   │  LLM 生成     │
                                   │  引用溯源      │
                                   └──────────────┘
```

### 向量数据库选型

| 数据库 | QPS(百万768D) | P99延迟 | 最大规模 | 推荐场景 |
|--------|-------------|---------|---------|---------|
| **Milvus** | 10,000 | 50ms | 十亿+级 | 超大规模生产环境 |
| **Qdrant** | 12,000 | 18ms | 亿级 | 极致低延迟 |
| **Weaviate** | 8,500 | 35ms | 千万~亿级 | 混合搜索(BM25+向量) |
| **pgvector** | 1,200 | 220ms | 千万级 | 已有 PG 生态/零运维 |
| **Chroma** | 800 | 150ms | 百万级 | 原型开发/小规模 |

> **选型建议**：已有 PostgreSQL 基础设施且数据量 < 千万 → pgvector；追求极致性能 → Qdrant；十亿级数据 → Milvus。

### 推理框架选型

| 框架 | 核心优势 | 吞吐量(7B) | 适用场景 |
|------|---------|-----------|---------|
| **vLLM** | PagedAttention + 动态批处理 | 800-1000 TPS | 企业高并发服务首选 |
| **SGLang** | RadixAttention + 结构化输出 | 600-800 TPS | 多模态+复杂控制流 |
| **TensorRT-LLM** | NVIDIA 极致优化 | 极高(需编译) | NVIDIA GPU 追求极致低延迟 |
| **TGI** | HuggingFace 生态集成 | 高 | HF 模型快速部署 |
| **Ollama** | 零配置、一键启动 | 100-150 TPS | 个人/小团队/边缘部署 |
| **LMDeploy** | 国产芯片优化 | 400-600 TPS | 昇腾/海光等国产化场景 |

> **选型决策**：企业级高并发 → vLLM；强制国产化 → LMDeploy；个人/低资源 → Ollama；NVIDIA 极致优化 → TensorRT-LLM。

## 四、模型选型：选对了事半功倍

### 主流开源模型对比

| 模型 | 参数规模 | 许可证 | 中文能力 | 代码能力 | 核心优势 |

| **Qwen2.5** | 0.5B-110B | Apache 2.0 | ★★★★★ | ★★★★ | 中文最强，尺寸最全，商用友好 |

## 五、部署步骤：从零到上线的完整路径

### 第一阶段：需求评估（1-2周）

| 任务 | 交付物 | 关键决策 |
|------|-------|---------|
| 明确业务场景 | 场景清单 | 问答？生成？分析？ |
| 数据盘点 | 数据资产表 | 有多少文档？什么格式？质量如何？ |
| 安全合规评估 | 合规报告 | 数据能否出域？是否需要信创？ |
| 预算评估 | 预算方案 | 硬件预算？人力预算？运维预算？ |

### 第二阶段：技术验证（2-4周）

**Step 1：搭建最小可用系统**

```bash
# 一行命令启动 7B 模型（Ollama，适合快速验证）
ollama run qwen2.5:7b

# 企业级推理服务（vLLM，适合生产部署）
python -m vllm.entrypoints.openai.api_server \
  --model Qwen/Qwen2.5-7B-Instruct \
  --tensor-parallel-size 1 \
  --gpu-memory-utilization 0.9 \
  --max-model-len 8192
```

**Step 2：搭建 RAG 知识库**

```bash
# 向量数据库（以 Milvus 为例）
docker run -d --name milvus \
  -p 19530:19530 -p 9091:9091 \
  milvusdb/milvus:v2.4-latest

# 文档导入 + Embedding + 检索（以 LangChain 为例）
# 1. 文档切分 → 2. Embedding → 3. 存入向量库 → 4. 检索+生成
```

**Step 3：评估效果**

| 评估维度 | 指标 | 及格线 |
|---------|------|-------|
| 回答准确率 | 人工评估正确率 | > 85% |
| 检索命中率 | Top-5 召回率 | > 90% |
| 响应延迟 | 首 Token 时间 | < 2s |
| 并发能力 | QPS | 满足业务峰值 |

## 七、分发形式：不同场景不同打法

| 分发形式 | 硬件要求 | 模型规模 | 典型延迟 | 典型场景 |
|---------|---------|---------|---------|---------|
| **SaaS/API** | 无 | 千亿~万亿 | 200ms-2s | 快速验证、通用任务 |
| **私有化部署** | GPU集群 | 7B-671B | 50-500ms | 金融、政务、医疗 |
| **边缘部署** | 工控机+GPU | 7B-14B(量化) | 30-200ms | 工厂、门店、离线场景 |
| **端侧(PC)** | 本地GPU/CPU | 7B-14B(量化) | 50-500ms | 本地Copilot、离线办公 |
| **端侧(手机)** | NPU | 0.5B-4B | 10-100ms | 手机AI助手、隐私场景 |

### 边缘与端侧部署的关键技术

- **量化**：INT4/INT8/FP8 降低显存 4-8 倍，精度损失可控
- **蒸馏/剪枝**：用大模型蒸馏出小模型，保留核心能力
- **主流端侧小模型**：Qwen2.5-0.5B/1.5B/4B、Llama 3.2 1B/3B、Phi-4
- **加速框架**：OpenVINO(Intel)、MLX(Apple Silicon)、NNAPI(Android)

> **工业场景的典型架构**：云端部署 72B 大模型做知识问答和文档生成，边缘侧部署 7B 量化模型做实时质检辅助和设备故障诊断——大小模型协同，云边端联动。

## 八、企业案例

### 案例一：某制造企业——工艺知识助手

**背景**：一家大型制造企业，工艺文档分散在 Word/PDF/Excel 中，新工程师上手慢，老师傅退休知识就流失。

**方案**：
- **模型**：Qwen2.5-14B（INT4量化，单卡 RTX 4090 即可运行）
- **架构**：vLLM + Milvus + LangChain
- **RAG**：将 5 万+ 份工艺文档切片→Embedding→存入向量库
- **效果**：工艺查询响应时间从平均 2 小时缩短到 10 秒，准确率 > 90%

**成本**：硬件 ¥3万 + 部署开发 ¥10万 + 年运维 ¥5万 ≈ **首年 ¥18万**

### 案例二：某金融机构——合规审查助手

**背景**：某银行需要对海量合同、监管文件进行合规审查，人工审查效率低、一致性差。

**方案**：
- **模型**：Qwen2.5-72B（私有化部署，4×A100）
- **架构**：vLLM + pgvector + Dify 编排平台
- **RAG**：法规库 + 内部合规手册 + 历史审查案例
- **效果**：审查效率提升 60%，合规风险识别准确率 > 95%

**成本**：硬件 ¥60万 + 部署开发 ¥30万 + 年运维 ¥30万 ≈ **首年 ¥120万**

### 案例三：某央企——工业设备智能诊断

**背景**：国家能源集团，设备种类多、故障模式复杂，运维人员经验不足。

**方案**：
- **模型**：私有化部署大模型 + 知识库
- **架构**：RAG + 设备故障知识库 + 实时数据接入
- **效果**：国内首个工业设备综合诊断运维大模型，故障诊断效率显著提升

**分发形式**：云端统一训练 → 边缘侧推理（工厂端部署 7B 量化模型，实时响应）

| # | 建议 | 说明 |
|---|------|------|
| 1 | **先调 API 验证，再考虑私有化** | 不要一上来就买 GPU，先用 API 跑通业务逻辑 |
| 2 | **7B 模型 + RAG 解决 80% 的问题** | 大多数企业场景不需要百亿参数，RAG 比更大的模型更有价值 |
| 3 | **数据质量决定上限** | 再好的模型配垃圾数据也是垃圾，先花精力整理知识库 |
| 4 | **量化是降本利器** | INT4 量化让 7B 模型只需 4GB 显存，单卡 4090 可以跑 14B |
| 5 | **vLLM 是企业部署首选** | PagedAttention + 动态批处理，吞吐量是原始推理的 3-5 倍 |
| 6 | **国产化路线提前规划** | 如果有信创要求，一开始就选昇腾 + DeepSeek，避免后期迁移 |
| 7 | **监控比模型更重要** | 上线后首 Token 延迟、错误率、GPU 利用率是核心指标 |
| 8 | **知识库需要持续更新** | 文档更新→重索引的自动化流程是长期运营的关键 |
| 9 | **混合方案性价比最高** | 核心场景私有化 + 通用能力 API，兼顾安全和成本 |
| 10 | **从最痛的场景切入** | 不要一上来就搞"全公司AI化"，先解决一个痛点，做出效果再推广 |

---

## 结语

企业大模型部署不是一个技术问题，而是一个**系统工程**——从业务需求到硬件选型，从架构设计到持续运维，每个环节都需要权衡成本、性能和安全。

回到最核心的判断：

- **有数据、有场景、有预算** → 私有化部署，7B + RAG 起步
- **有场景、没 GPU** → API 调用，DeepSeek 性价比最高
- **有信创要求** → 昇腾 + DeepSeek，当前最成熟国产化方案
- **有边缘需求** → 7B 量化模型 + Ollama，工控机即可运行

**最重要的一条：先跑起来，再优化。** 一个跑在 RTX 4090 上的 7B + RAG 系统，如果能解决一个真实的业务痛点，它的价值远超一个永远在评估中的百亿参数集群。

> 在工程领域，一个运行的简单方案，永远胜过一个完美的复杂方案。
