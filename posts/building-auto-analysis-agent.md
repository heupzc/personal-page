# 构建自动分析 Agent：架构方案、技术选型与大模型依赖的真相

> "我想做一个自动分析的 Agent，但不知道该用什么框架、什么架构，是不是必须接 GPT 或 Claude？"——这是 2025 年开发者问得最多的问题之一。本文系统梳理 Agent 的架构模式、框架选型、大模型依赖的真相，以及不同类型 Agent 的构建差异。

---

## 一、Agent 是什么？先搞清楚边界

### Agent ≠ 聊天机器人

很多人把 Agent 等同于"加了工具调用的 ChatGPT"，这是不对的。两者的本质区别：

| 维度 | 聊天机器人 | Agent |
|------|-----------|-------|
| **驱动方式** | 用户提问→模型回答 | 目标驱动→自主规划→执行→反馈 |
| **是否自主** | 被动响应 | 主动执行多步骤任务 |
| **是否记忆** | 通常无状态 | 有记忆、有状态、可回溯 |
| **是否纠错** | 不会自我纠正 | 可以反思、回退、重试 |
| **典型场景** | "帮我翻译这段话" | "帮我分析这个月的销售数据，生成报告，异常点深入挖掘" |

> **一句话定义**：Agent 是一个**目标驱动的自主系统**——给定一个目标，它能自主拆解任务、调用工具、处理异常、交付结果。

## 二、Agent 的四种核心架构模式

### 模式一：ReAct（推理+行动循环）

ReAct 是最经典的 Agent 模式，核心思想是**交替推理和行动**：

```
用户目标
   │
   ▼
┌─────────┐    ┌─────────┐    ┌─────────┐
│ Thought │───→│ Action  │───→│Observation│
│ "我需要  │    │ 调用工具 │    │ 工具返回  │
│ 先查数据"│    │ 查询数据库│    │ 查询结果  │
└─────────┘    └─────────┘    └────┬────┘
                                   │
                              ┌────▼────┐
                              │ 再思考   │
                              │ "数据拿到│
                              │ 了，需要 │
                              │ 分析趋势"│
                              └────┬────┘
                                   │
                              ┌────▼────┐
                              │ Thought │───→ Action ───→ Observation ───→ ...
                              └─────────┘
                                   │
                              达成目标，输出结果
```

**优点**：简单直观、可解释性强（每一步都能看到推理过程）
**缺点**：每步都调模型，Token消耗大；长链路容易"跑偏"
**适用**：简单工具调用、单步推理场景

### 模式二：Plan-and-Execute（先规划后执行）

与 ReAct 的"边想边做"不同，Plan-and-Execute 是**先做全局规划，再逐步执行**：

```
用户目标
   │
   ▼
┌──────────────────────┐
│       规划器          │
│  生成完整执行计划：     │
│  Step1: 查销售数据    │
│  Step2: 计算环比      │
│  Step3: 检测异常      │
│  Step4: 生成报告      │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│       执行器          │
│  按计划逐步执行        │
│  Step1 → Step2 → ... │
│  如果某步失败，        │
│  反馈给规划器重新规划   │
└──────────────────────┘
```

**优点**：全局视角更优、避免"走一步看一步"的盲目性
**缺点**：规划可能不完整，需要重新规划机制
**适用**：多步骤复杂任务、需要全局视角的分析任务

### 模式三：Reflection（自我反思）

在 ReAct 或 Plan-Execute 基础上增加**自我评估和纠错**环节：

```
执行结果
   │
   ▼
┌──────────┐     不满意     ┌──────────┐
│  评估器   │─────────────→│  修正器   │
│  结果OK?  │              │  重新执行  │
└────┬─────┘              └──────────┘
     │ 满意
     ▼
  输出最终结果
```

**优点**：结果质量更高、具备自我纠错能力
**缺点**：额外的模型调用成本、可能陷入"反思死循环"
**适用**：对结果质量要求高的场景（代码生成、研究报告）

### 模式四：Multi-Agent（多智能体协作）

把一个复杂任务拆分给多个专业 Agent 协作完成：

```
用户目标
   │
   ▼
┌──────────┐
│  编排Agent │  负责任务分发和结果汇总
└──┬───┬───┘
   │   │   │
   ▼   ▼   ▼
┌────┐┌────┐┌────┐
│调研 ││分析 ││写作 │  各自专业，并行工作
│Agent││Agent││Agent│
└──┬─┘└──┬─┘└──┬─┘
   │      │      │
   └──────┼──────┘
          ▼
     汇总输出结果
```

**优点**：每个 Agent 专注一件事，做得更好；可并行提升效率
**缺点**：协调成本高、调试困难、Agent 间信息传递可能丢失
**适用**：复杂的多步骤任务、需要多种专业能力的场景

### 四种模式对比

| 维度 | ReAct | Plan-Execute | Reflection | Multi-Agent |
|------|-------|-------------|-----------|-------------|
| **复杂度** | ⭐ | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| **可控性** | 低 | 高 | 中 | 中 |
| **Token消耗** | 高 | 中 | 很高 | 很高 |
| **结果质量** | 中 | 中高 | 高 | 高 |
| **调试难度** | 低 | 中 | 中 | 高 |
| **适用场景** | 简单工具调用 | 多步分析任务 | 高质量输出 | 复杂协作任务 |

> **实践建议**：从 ReAct 起步，简单场景够用；需要全局规划升级 Plan-Execute；对质量有要求加 Reflection；多专业协作才上 Multi-Agent。不要一上来就搞最复杂的。

## 三、框架选型：七大 Agent 框架对比

### 一张表看全

| 框架 | 核心理念 | 学习曲线 | 生产就绪 | 适用场景 |
|------|---------|---------|---------|---------|
| **LangGraph** | 图结构+状态管理 | ⭐⭐⭐⭐⭐ | ★★★★★ | 复杂流程、生产级系统 |
| **CrewAI** | 团队式角色协作 | ⭐⭐ | ★★★★ | 流程自动化、内容管线 |
| **AutoGen** | 多Agent对话 | ⭐ | ★★★ | 快速原型、学术研究 |
| **Dify** | 低代码可视化 | ⭐ | ★★★★ | 非开发者、快速搭建 |
| **OpenAI Agents SDK** | 轻量级、OpenAI生态 | ⭐⭐ | ★★★ | OpenAI用户、简单Agent |
| **Semantic Kernel** | 企业级、.NET/Java | ⭐⭐⭐ | ★★★★★ | 微软生态企业 |
| **自研** | 完全定制 | ⭐⭐⭐⭐⭐ | 看实力 | 特殊需求、极致优化 |

### 选型决策树

```
你的团队有专职开发者吗？
│
├── 没有 → Dify（低代码，可视化搭建）
│
├── 有，但人少，想快速出活
│   ├── 场景是团队协作/流程自动化 → CrewAI
│   └── 场景是快速验证概念 → AutoGen
│
├── 有，且需要生产级系统
│   ├── 复杂流程+状态管理 → LangGraph
│   └── 微软生态企业 → Semantic Kernel
│
└── 有，且需求特殊/极致优化 → 自研
    └── 基于开源组件组装：推理引擎+工具调度+记忆+监控
```

### LangGraph vs CrewAI vs AutoGen：三大框架深度对比

| 维度 | LangGraph | CrewAI | AutoGen |
|------|-----------|--------|---------|
| **架构思想** | 图（Graph）：节点+边 | 团队（Crew）：角色+任务 | 对话（Chat）：Agent间交流 |
| **流程控制** | 显式定义流程图 | YAML配置任务依赖 | 隐式，靠对话驱动 |
| **状态管理** | 内置状态机 | 弱 | 弱，长流程容易"忘" |
| **可观测性** | 强（LangSmith集成） | 中 | 弱 |
| **适合谁** | 架构师/平台开发者 | 中小团队/企业 | 个人开发者/研究者 |

> **一句话总结**：AutoGen 快、CrewAI 稳、LangGraph 强。

## 四、大模型依赖的真相：Agent 必须依赖 Claude/GPT 吗？

### 答案：不是必须的，但取决于你要做什么

很多人以为 Agent = 大模型 + 工具调用，所以必须依赖 GPT/Claude。但实际情况更复杂：

```
Agent 的大模型依赖程度光谱

完全不需要                    完全依赖
    │                           │
    ├── 规则驱动Agent ──────────┤
    │   (工作流引擎、n8n)       │
    │                          │
    ├── 混合Agent ─────────────┤
    │   (规则+小模型)           │
    │                          │
    ├── 轻量LLM Agent ─────────┤
    │   (开源7B模型+工具)       │
    │                          │
    └── 重量LLM Agent ─────────┤
        (GPT-4/Claude+复杂工具) │
```

### 四种依赖程度，四种构建方式

#### 1. 规则驱动 Agent：完全不依赖大模型

**原理**：用预设的规则和流程驱动任务执行，类似传统的工作流引擎。

```
触发条件 → 规则匹配 → 执行动作 → 检查结果 → 下一步
```

**技术栈**：n8n / Apache Airflow / Prefect / 自研规则引擎

**适用场景**：
- 流程固定、规则明确的数据ETL
- 定时报表生成和分发
- 监控告警→自动处置

**优点**：零Token成本、确定性强、可解释
**缺点**：无法处理未预见的异常、没有"理解"能力

#### 2. 混合 Agent：规则 + 小模型

**原理**：规则处理确定性流程，小模型处理模糊判断。

```
┌──────────────────────────────────────┐
│             编排层（规则引擎）          │
│   Step1: 数据获取（规则）             │
│   Step2: 异常检测（规则+小模型）       │
│   Step3: 根因分析（小模型）           │
│   Step4: 报告生成（模板+小模型填充）   │
│   Step5: 结果分发（规则）             │
└──────────────────────────────────────┘
```

**技术栈**：n8n/LangChain + 本地7B模型（Qwen2.5-7B/Ollama）

**适用场景**：
- 工业数据分析（异常检测+根因分析）
- 金融风控（规则筛+模型判）
- 合规审查（规则检查+模型理解语义）

**优点**：低成本、低延迟、数据不出域
**缺点**：小模型的推理能力有限，复杂分析可能不够

#### 3. 轻量 LLM Agent：开源模型 + 工具

**原理**：用开源模型（7B-14B）做推理，结合工具调用完成分析任务。

**技术栈**：Ollama/vLLM + Qwen2.5-14B + 工具集

**适用场景**：
- 内部知识库问答
- 代码辅助分析
- 文档摘要和分类

**优点**：完全私有化、中文能力强、可定制
**缺点**：复杂推理不如大模型、工具调用格式可能不稳定

#### 4. 重量 LLM Agent：GPT-4/Claude + 复杂工具链

**原理**：用最强大模型做推理核心，配合丰富的工具生态。

**技术栈**：Claude/GPT-4 + MCP/LangGraph + 完整工具链

**适用场景**：
- 复杂研究调研
- 多步代码生成和调试
- 多Agent协作的复杂系统

**优点**：推理能力最强、工具调用最稳定、生态最丰富
**缺点**：成本高、数据出境、依赖第三方

### 四种方式的成本对比

| 方式 | 硬件成本 | Token成本/月 | 延迟 | 推理能力 |
|------|---------|------------|------|---------|
| 规则驱动 | ¥0 | ¥0 | 毫秒 | ★☆☆☆☆ |
| 混合Agent | ¥1-3万(1×4090) | ¥0 | 秒级 | ★★★☆☆ |
| 轻量LLM | ¥1-3万(1×4090) | ¥0 | 秒级 | ★★★★☆ |
| 重量LLM | ¥0(调API) | ¥0.5-5万 | 2-5秒 | ★★★★★ |

> **核心结论**：Agent 不等于大模型。如果你的场景规则明确，规则驱动就够了；如果需要"理解"但预算有限，混合方案是最佳平衡点；只有需要强推理的场景才必须上大模型。

## 五、工具调用：MCP vs Function Calling

### Function Calling：模型的"表达能力"

Function Calling 是 LLM 输出结构化工具调用指令的能力——模型决定调用什么函数、传什么参数，但**不负责执行**。

```
用户 → LLM → 输出 {"function": "query_db", "args": {"sql": "SELECT ..."}}
                        │
                        ▼
              开发者写代码执行这个函数
                        │
                        ▼
              结果返回给 LLM → 生成回答
```

**特点**：
- 与特定平台绑定（OpenAI的格式、Claude的格式各不同）
- 每增加一个功能需要改代码、重新定义 Schema
- 工具定义和模型耦合在一起

### MCP：AI 应用的"集成标准"

MCP（Model Context Protocol）是一套开放协议标准，定义了 AI 应用如何与外部工具/数据源交互。

```
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│   MCP Host   │      │  MCP Client  │      │  MCP Server  │
│  (宿主应用)   │──────│  (协议客户端)  │──────│  (工具提供方)  │
│              │      │  1:1连接管理   │      │  暴露能力/资源  │
│  Claude Desk │      │              │      │  数据库/文件/API│
│  IDE/Cursor  │      │              │      │              │
└──────────────┘      └──────────────┘      └──────────────┘
```

**MCP 的三大能力**：
- 🔧 **Tools**：让 LLM 调用外部操作（数据库查询、API调用、文件操作）
- 📄 **Resources**：向 LLM 提供上下文数据（文件内容、数据库记录）
- 💬 **Prompts**：预定义的提示词模板，标准化交互

### 核心区别

| 维度 | Function Calling | MCP |
|------|-----------------|-----|
| **本质** | LLM 的输出能力 | 开放协议标准 |
| **耦合度** | 与特定平台/模型绑定 | 模型无关、平台无关 |
| **扩展性** | 每增功能需改代码 | 插件式热插拔，动态发现 |
| **通信方式** | 请求-响应，一次性 | 持久连接，双向通信 |
| **安全控制** | 依赖应用层实现 | 协议内置权限与沙箱 |
| **生态** | 各厂商各自定义 | 统一标准，跨应用复用 |
| **成熟度** | ★★★★★ 生产可用 | ★★★☆☆ 快速发展中 |

> **选型建议**：如果是单一模型、工具少 → Function Calling 足够；如果多模型切换、工具生态丰富、需要标准化 → MCP 是未来方向。**但 MCP 目前仍在快速迭代中，生产环境需谨慎评估。**

## 六、不同类型 Agent 的构建差异

### 数据分析 Agent

**典型任务**：连接数据库→SQL查询→数据分析→图表生成→报告输出

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| 意图理解 | LLM (7B-14B够用) | 把自然语言转为SQL查询意图 |
| SQL生成 | LLM + Schema提示 | 需要数据库Schema作为上下文 |
| 数据获取 | 数据库连接池 | MySQL/PostgreSQL/DuckDB |
| 分析推理 | LLM / 规则引擎 | 统计分析可以用规则，异常检测用模型 |
| 可视化 | Matplotlib/ECharts/Plotly | 根据分析结果自动选图表类型 |
| 报告生成 | LLM + 模板 | 数据填充+文字总结 |

**架构推荐**：Plan-Execute（先规划分析步骤，再逐步执行）

**关键难点**：SQL生成的准确性——需要把完整的Schema信息注入Prompt，包括表结构、字段含义、常见查询模式

### 研究调研 Agent

**典型任务**：搜索资料→阅读提炼→交叉验证→生成报告

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| 搜索 | Web搜索API | Tavily/Serper/Brave Search |
| 网页解析 | HTML→Markdown | Readability/Jina Reader |
| 长文阅读 | 长上下文模型 | 128K+上下文窗口 |
| 信息提炼 | LLM | 摘要、关键信息提取 |
| 交叉验证 | LLM + 事实核查 | 多源比对、矛盾检测 |
| 报告生成 | LLM | 综合分析+结构化输出 |

**架构推荐**：Multi-Agent（搜索Agent+阅读Agent+写作Agent协作）

**关键难点**：信息可靠性——需要多源验证，避免"一个信源出错全部出错"

### 代码生成 Agent

**典型任务**：理解需求→设计方案→编码→测试→调试→提交

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| 需求理解 | 强推理LLM | 需要理解模糊需求并澄清 |
| 代码生成 | 代码LLM | Claude/GPT-4/DeepSeek |
| 代码执行 | 沙箱环境 | Docker/VM隔离执行 |
| 测试 | 测试框架 | 自动生成测试用例并运行 |
| 调试 | LLM + 错误日志 | 分析报错、定位问题、修复 |

**架构推荐**：ReAct + Reflection（写代码→运行→看报错→修复→再运行）

**关键难点**：调试循环——需要沙箱安全执行、正确解析错误信息、避免"改一个Bug引入三个"

### 工作流自动化 Agent

**典型任务**：接收工单→判断类型→调用系统→执行操作→反馈结果

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| 意图分类 | 小模型/规则 | 工单类型识别 |
| 系统调用 | API/MCP | ERP/MES/OA系统操作 |
| 流程控制 | 规则引擎 | 确定性流程用规则 |
| 异常处理 | LLM | 非标准情况用模型判断 |
| 人工审批 | 通知+审批流 | 高风险操作需人工确认 |

**架构推荐**：混合Agent（规则+模型），**不建议纯LLM驱动**

**关键难点**：可靠性——工业流程中操作错误可能导致停线，必须规则兜底+人工审批

### 四种 Agent 对比

| 维度 | 数据分析 | 研究调研 | 代码生成 | 工作流自动化 |
|------|---------|---------|---------|------------|
| **模型依赖** | 中（7B-14B够用） | 高（需要长上下文） | 高（需要强推理） | 低（规则为主） |
| **架构推荐** | Plan-Execute | Multi-Agent | ReAct+Reflection | 混合（规则+模型） |
| **工具复杂度** | 中 | 低 | 高（沙箱+CI） | 高（多系统集成） |
| **对准确性要求** | 高 | 中 | 高 | 极高 |
| **是否可离线** | 可以 | 不行（需联网） | 可以 | 可以 |
| **典型框架** | LangGraph | CrewAI | LangGraph | n8n+轻量模型 |

## 七、从零构建一个数据分析 Agent：实战示例

### 最小可用版本（30分钟搭完）

```python
import json
from openai import OpenAI

client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")
# 使用本地 Ollama + Qwen2.5-7B，完全不依赖外部API

# 工具定义
tools = [
    {
        "type": "function",
        "function": {
            "name": "query_database",
            "description": "查询数据库，执行SQL语句",
            "parameters": {
                "type": "object",
                "properties": {
                    "sql": {"type": "string", "description": "SQL查询语句"}
                },
                "required": ["sql"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "plot_chart",
            "description": "生成图表",
            "parameters": {
                "type": "object",
                "properties": {
                    "chart_type": {"type": "string", "enum": ["bar", "line", "pie"]},
                    "data": {"type": "string", "description": "图表数据JSON"},
                    "title": {"type": "string", "description": "图表标题"}
                },
                "required": ["chart_type", "data", "title"]
            }
        }
    }
]

# ReAct 循环
def run_agent(user_query: str, max_steps: int = 5):
    messages = [{"role": "user", "content": user_query}]
    
    for step in range(max_steps):
        response = client.chat.completions.create(
            model="qwen2.5:7b",
            messages=messages,
            tools=tools
        )
        
        msg = response.choices[0].message
        
        # 如果模型直接返回文字，说明任务完成
        if msg.content and not msg.tool_calls:
            return msg.content
        
        # 如果模型调用工具，执行并返回结果
        if msg.tool_calls:
            for tool_call in msg.tool_calls:
                result = execute_tool(tool_call.function.name, 
                                     json.loads(tool_call.function.arguments))
                messages.append({"role": "tool", 
                               "content": json.dumps(result),
                               "tool_call_id": tool_call.id})
    
    return "达到最大步数限制，任务未完成"

def execute_tool(name: str, args: dict):
    if name == "query_database":
        # 实际项目中连接数据库执行SQL
        return {"data": "查询结果..."}
    elif name == "plot_chart":
        # 实际项目中生成图表
        return {"chart_url": "charts/output.png"}
```

### 生产级版本的关键增强

| 增强项 | 说明 | 技术方案 |
|--------|------|---------|
| **SQL安全** | 防止注入、只允许SELECT | SQL解析器+白名单校验 |
| **Schema注入** | 让模型知道数据库结构 | 把DDL和字段说明注入系统提示 |
| **结果缓存** | 相同查询不重复执行 | Redis缓存层 |
| **分析规划** | 先规划再执行 | Plan-Execute替代纯ReAct |
| **自我校验** | SQL执行前先校验语法 | 语法检查+Dry Run |
| **可视化推荐** | 自动选最佳图表类型 | 基于数据特征推荐 |
| **多轮交互** | 不确定时追问用户 | 澄清机制 |
| **审计日志** | 每一步操作可追溯 | 日志系统 |

## 八、十条实践建议

| # | 建议 | 说明 |
|---|------|------|
| 1 | **从最小可用版本起步** | 不要一上来就搞 Multi-Agent，先跑通 ReAct 单步调用 |
| 2 | **先用 API 验证，再换本地模型** | 先用 GPT/Claude 验证架构可行，再替换为开源模型降本 |
| 3 | **规则能解决的不用模型** | 确定性流程用规则，模糊判断才用模型——混合方案最务实 |
| 4 | **工具定义比模型更重要** | 好的工具定义（清晰的描述、合理的参数）比换更强的模型更有效 |
| 5 | **加 Reflection 提升质量** | 花额外 Token 做自检，比一次性生成质量高得多 |
| 6 | **MCP 是趋势但不是必须** | Function Calling 够用就别追 MCP，等生态成熟再迁移 |
| 7 | **7B 模型做工具调用已够用** | Qwen2.5-7B 的工具调用能力已经相当不错，不必上来就用 72B |
| 8 | **安全兜底比智能更重要** | Agent 可能误操作，必须加权限控制、人工审批、操作回滚 |
| 9 | **可观测性是上线的前提** | 每步推理、每次工具调用都要有日志，否则出了问题无法排查 |
| 10 | **不要为了用 Agent 而用 Agent** | 如果一个简单的 if-else 就能解决，不需要 Agent |

## 结语

构建自动分析 Agent，核心不是选哪个框架、接哪个模型，而是**想清楚你的场景到底需要什么程度的"智能"**。

- 规则明确 → 规则引擎，不依赖大模型
- 需要理解语义 → 混合方案，规则+小模型
- 需要推理分析 → 轻量LLM，7B-14B够用
- 需要强推理+复杂工具 → 重量LLM，GPT-4/Claude

Agent 的本质不是"大模型+工具"，而是**一个能自主完成任务的系统**。大模型只是这个系统的推理引擎——你完全可以用规则引擎、小模型甚至硬编码来替代它，只要能完成目标。

**选最简单的方案，解决最痛的问题。** 其他的，等跑起来再说。

> The best Agent is the one that solves your problem, not the one with the most parameters.
